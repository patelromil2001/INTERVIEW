🩺 Loop Health — Claim Support Tool
BigQueryReactNode.jsTPA APIsObservability
How to Explain It (30-second pitch)
Built a full-stack internal tool for a health-insurance platform that automated claim status tracking, reducing support agent workload by ~9,000 hours per month.
Integrated with 10+ Third Party Administrators (TPAs) via APIs to pull real-time claim data, which cut inbound support tickets by 50%.
Set up an observability layer using BigQuery to log events, making debugging significantly faster.
Worked across 8 concurrent feature streams simultaneously — meaning you had to manage priorities, avoid breaking shared code, and coordinate merges.
Likely Interview Questions
How did you calculate the "9,000 agent hours saved"?
Measure pre-launch ticket volume × avg handle time, then compare post-launch. If ~5,000 tickets/month at ~1.8 hrs each, that's 9,000 hrs. Be specific about the baseline and methodology.
What is a TPA and how did you integrate with them?
Third Party Administrators manage insurance claims on behalf of insurers. Integration was via REST APIs with authentication tokens, polling claim statuses and mapping their data schemas to the internal model.
Why BigQuery for observability instead of something like Datadog?
BigQuery allows SQL-based ad hoc analysis over large event volumes at low cost. It's great when the team already uses GCP and wants to run custom queries over historical logs without a separate APM budget.
How did you manage 8 concurrent projects without conflicts?
Feature branches per project, regular rebasing from main, clear module separation to avoid cross-project collisions, and daily standups to surface blockers early.
📚 Bookshelf Web App
Next.jsTailwindshadcnPlotlyGoogle Books APIVercelGitHub Actions
How to Explain It
A book management and discovery platform that pulls data from the Google Books API for search, metadata, and covers.
Includes data visualization via Plotly (e.g. reading stats, genre breakdowns) and potentially prediction features.
Built with Next.js App Router, Tailwind for styling, and shadcn/ui component library for consistent design system.
Deployed on Vercel with automated CI/CD via GitHub Actions — every push to main triggers a deployment pipeline.
Likely Interview Questions
Why Next.js over plain React?
Server-side rendering (SSR) and Static Site Generation (SSG) for faster initial loads and better SEO. Also built-in routing (App Router), API routes, and image optimization out of the box.
What is shadcn/ui and why use it over other component libraries?
shadcn/ui is not a typical npm package — it's a collection of copy-paste components built on Radix UI primitives and styled with Tailwind. You own the code, can customize freely, no version lock-in. Better for projects needing design control.
Walk me through your GitHub Actions CI/CD workflow
On push to main: 1) Checkout code → 2) Install deps → 3) Run lint + tests → 4) Build → 5) Deploy to Vercel via Vercel CLI or Vercel GitHub integration. Failures at any step block the merge.
✈️ Wory App — Aviation Tool
ReactMaterial UIContext APIPythonFastAPIGraphQLDockerAWS SES/S3
How to Explain It
A flight-data web application integrating third-party aviation APIs (likely AviationStack or FlightAware) to track and process flight datasets.
Backend is Python + FastAPI with a GraphQL layer — meaning clients can query exactly the data they need rather than over-fetching.
AWS SES for transactional emails (notifications, alerts), S3 for storing datasets or user assets, CodePipeline for automated builds.
Docker containerizes the backend ensuring consistent environments from dev to production.
Likely Interview Questions
REST vs GraphQL — why did you choose GraphQL here?
GraphQL lets clients specify exactly which fields they need, reducing over-fetching. Ideal for flight data where different consumers (mobile, web, stakeholder APIs) need different data shapes. REST would require multiple endpoints or send excess data.
What is Context API and when would you use Redux instead?
Context API is React's built-in solution for sharing state without prop drilling — great for simple/moderate state like user auth or theme. Use Redux when you need complex state logic, time-travel debugging, or very large apps where performance optimization (selectors) matters.
How does AWS SES work?
Simple Email Service — send transactional emails programmatically. You verify a domain, call the SES API (or SDK) with recipient/subject/body, and AWS handles delivery. Used here for sending flight alerts or status notifications to users.
